2. Purpose & Scope

Purpose of the Document

This document defines the technical architecture, data flow, database design, and security controls for a centralized License Authority & Key Management System (LAM-KMS) using RSA public/private key cryptography for fintech microservices.

Scope of the System

  • License enforcement for backend microservices
  • Daily RSA key rotation and distribution
  • Kill switch and grace-period handling

In-Scope Components

  • License Authority Microservice
  • RSA Key Management (Public/Private)
  • Redis License Cache
  • Project Microservices (Laravel)

Out-of-Scope Components

  • Customer authentication
  • Transaction business logic

Assumptions & Dependencies

  • All service communication uses TLS 1.2+
  • Redis is protected using ACL and AUTH
  • Private RSA keys never leave LAM/KMS

4.4 License Diagnostic & Tamper Detection Framework

4.4.1 Objective

This section defines the diagnostic, integrity validation, and tamper detection mechanisms that ensure License Authority & KMS (LAM) maintains centralized control over all licensed microservices, even when deployed on client-managed infrastructure.

  • Detect unauthorized code or configuration changes
  • Prevent license bypass or replay attacks
  • Ensure kill-switch authority remains with LAM
  • Provide audit-grade forensic evidence

4.4.2 Diagnostic Control Principles

Principle Description
Centralized Authority LAM is the single source of truth for license validity
Zero Trust Execution Microservices never self-assert license state
Cryptographic Binding License is bound to service identity, code state, and host
Time-Bound Trust All licenses expire and require revalidation
Detect Over Prevent System prioritizes detection and controlled shutdown

4.4.3 License Assertion Token (LAT)

LAM issues a License Assertion Token (LAT), cryptographically signed using the LAM private RSA key. The LAT represents the authoritative license state.

LAT Properties

  • RSA signed (SHA-256)
  • Short-lived and non-replayable
  • Verifiable but non-modifiable by clients

LAT Payload Structure

{
  "project_id": "FINTECH_X",
  "service_name": "payments-service",
  "service_hash": "sha256_code_manifest",
  "host_fingerprint": "hashed_machine_identity",
  "key_version": "v7",
  "issued_at": "2026-01-14T10:00:00Z",
  "expires_at": "2026-01-15T10:00:00Z",
  "license_status": "ACTIVE"
}

Microservices can only verify the LAT using the public key fetched from LAM.

4.4.4 Runtime Code Integrity Diagnosis

Each microservice performs runtime integrity checks to detect unauthorized code changes.

Build-Time Manifest Generation

sha256sum app/ bootstrap/ vendor/ > service.manifest

Runtime Validation

  • Hash recalculated during startup and periodic intervals
  • Compared against service_hash in LAT
  • Mismatch indicates potential tampering
Integrity State Action
Hash Match Normal request processing
Temporary Mismatch Grace Mode + Alert to LAM
Persistent Mismatch License revocation and kill switch activation

4.4.5 Host Environment Fingerprinting

To prevent license reuse across unauthorized environments, each service derives a host fingerprint.

Fingerprint Inputs

  • CPU identifier
  • Primary disk UUID
  • MAC address

The combined fingerprint is hashed and embedded in the LAT. Any mismatch invalidates the license.

4.4.6 Heartbeat & Behavioral Diagnosis

Each microservice periodically sends a diagnostic heartbeat to LAM.

Heartbeat Payload

{
  "service_name": "payments-service",
  "lat_version": "v7",
  "license_state": "ACTIVE",
  "redis_hit_ratio": 98,
  "failed_license_checks": 0,
  "uptime_seconds": 86400
}

Behavioral Anomalies Detected

  • Missing or delayed heartbeats
  • Repeated grace-mode activations
  • Redis-only validation patterns
  • License refresh suppression attempts

4.4.7 Redis Trust Degradation Model

Redis is treated strictly as a performance cache, not a trust source.

  • All Redis license entries include RSA signatures
  • Signature verified on every read
  • Unsigned or invalid entries are ignored

4.4.8 Kill Switch Enforcement

The kill switch is data-driven and enforced through license state, not application code.

Kill Switch Triggers

  • License revoked
  • Persistent integrity failure
  • Project status set to SUSPENDED or REVOKED

If enforcement logic is removed from code, LAT renewal fails, resulting in automatic service shutdown on the next validation cycle.

4.4.9 Forensic Audit & Evidence Collection

LAM maintains immutable audit logs for all diagnostic events.

  • Code hash mismatches
  • Heartbeat anomalies
  • License issuance and revocation
  • Key rotation history

These logs provide regulatory, contractual, and legal evidence for compliance audits and client disputes.

4.5 Diagnostic Implementation Guidelines for All Microservices

4.5.1 Purpose

This section defines the mandatory implementation steps that every downstream microservice must follow to integrate with the License Authority & KMS (LAM). These controls ensure that license enforcement, diagnostics, and kill-switch authority remain centralized and tamper-resistant.

Non-compliance with any step in this section results in automatic license invalidation.

4.5.2 Mandatory Startup Diagnostic Sequence

Every microservice must execute the following sequence before accepting traffic: 

1. Load service identity (project_id, service_name)
2. Fetch active LAM public key via mTLS
3. Load last known License Assertion Token (LAT)
4. Verify LAT RSA signature
5. Validate LAT expiry and key_version
6. Compute runtime code hash
7. Validate host fingerprint
8. Load Redis cache (signed only)
9. Register heartbeat scheduler

Failure at any step prevents the service from starting or forces immediate shutdown.

4.5.3 License Middleware Enforcement

All request handling layers must be protected by a centralized License Middleware.

  • No controller, worker, job, or cron may bypass this middleware
  • Middleware validates LAT state on every request
  • Redis is consulted only after RSA verification

If middleware is disabled or removed, LAT renewal fails and the service enters kill-switch state.

4.5.4 License Assertion Token (LAT) Verification Rules

Microservices are strictly limited to verification-only operations on LAT.

  • LAT must be verified using the LAM public key
  • project_id and service_name must match runtime identity
  • key_version must be active
  • expires_at must be in the future

LAT modification, regeneration, or override is strictly forbidden.

4.5.5 Runtime Code Integrity Enforcement

Each microservice must continuously validate its own integrity.

  • Build-time manifest hash is generated during CI/CD
  • Runtime hash is computed at startup and at fixed intervals
  • Hash is compared against service_hash inside LAT
ConditionResult
Hash matchNormal operation
Single mismatchGrace Mode + LAM alert
Repeated mismatchImmediate kill switch

4.5.6 Host Fingerprint Validation

Host fingerprint validation prevents license reuse across machines or containers.

  • Fingerprint is derived from CPU ID, Disk UUID, and MAC
  • Fingerprint hash must match LAT
  • Any mismatch triggers immediate license revocation

Container restarts are allowed only if fingerprint remains consistent.

4.5.7 Redis Usage & Trust Model

Redis is treated as a non-authoritative cache.

  • All Redis entries must contain RSA signatures
  • Signature is verified on every read
  • Expired or unsigned entries are ignored

Redis availability issues must never disable license validation logic.

4.5.8 Heartbeat Scheduler Implementation

Each microservice must run a background heartbeat scheduler.

Heartbeat Responsibilities

  • Report license state
  • Report integrity status
  • Report grace-mode usage
  • Confirm Redis vs LAM validation ratio

Missing or delayed heartbeats are treated as behavioral anomalies.

4.5.9 Grace Mode vs Kill Switch Enforcement

Scenario Enforcement Action
LAM temporary unavailability Grace Mode (time-bound)
License expired Grace Mode + forced renewal
Code integrity failure Immediate Kill Switch
Host fingerprint mismatch Immediate Kill Switch
License revoked Immediate Kill Switch

4.5.10 Forbidden Developer Actions

  • Hardcoding license checks or flags
  • Mocking LAM in production
  • Bypassing license middleware
  • Extending LAT expiry locally
  • Suppressing heartbeat failures
  • Disabling integrity checks

Detection of any forbidden action results in permanent license revocation.

7. Non-Functional Requirements

  • Performance: Redis lookup < 10ms
  • Security: RSA 2048+, TLS 1.2+
  • Availability: Zero-downtime key rotation
  • Compliance: RBI, PCI-DSS aligned

8. Exception & Error Handling

  • License expired → Grace Mode
  • Project suspended → Immediate Kill Switch
  • LAM unavailable → Cached key usage

9. Deployment Specification

  • Dockerized Laravel services
  • Separate environment secrets
  • Rollback via Redis cache restore
